# Release process

The app is published via **GitHub Releases** and the auto-updater fetches
new versions from there. Two ways to ship a release:

## 1. Push a version tag (recommended)

```bash
# bump version in package.json (one of)
npm version patch    # 1.0.0 -> 1.0.1
npm version minor    # 1.0.0 -> 1.1.0
npm version major    # 1.0.0 -> 2.0.0

# commit + tag
git push origin main
git push origin v$(node -p "require('./package.json').version")
```

The GitHub Action at `.github/workflows/release.yml` triggers on any
`v*.*.*` tag, runs lint + typecheck + electron-builder, then publishes
the artifacts to a GitHub Release.

## 2. Manual trigger

In GitHub UI: **Actions → Release → Run workflow** → publishes whatever
build is on `main` using the version in `package.json`.

## Prerequisites

1. **Repo settings → Secrets** must have:
   - `GITHUB_TOKEN` (auto-provided, default permissions enough for
     `softprops/action-gh-release`).
2. **electron-builder.yml** `publish` block must point to the real
   repository:
   ```yaml
   publish:
     provider: github
     owner: <your-github-username>
     repo: robotrader-ai
   ```
3. **Tags must be reachable** by electron-updater at runtime. The
   `latest.yml` file generated by the build is what the updater
   actually fetches — make sure it's uploaded alongside the .exe.

## Code signing (optional but recommended)

Self-signed binaries trigger Windows SmartScreen. For silent updates,
buy an EV or OV code-signing certificate and add to electron-builder:

```yaml
win:
  certificateFile: build/cert.pfx
  certificatePassword: ${env.CSC_KEY_PASSWORD}
```

…then add the password as a GitHub secret and expose it in the workflow.
EV certificates also unlock reputation-based SmartScreen bypass on
first run.
